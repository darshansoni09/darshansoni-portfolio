Projects

Real-time Tracking & Monitoring Platform

When I joined the project, the system relied heavily on polling. Every device triggered API calls at fixed intervals even when there was no change in data.

This caused unnecessary load on the backend, slow map rendering, excessive database writes and increased infrastructure costs.

I redesigned the entire communication layer by replacing polling with an event-driven architecture using WebSockets and MQTT. Instead of constantly asking for updates, the system started reacting only when data actually changed.

I also optimized map performance by removing redundant state updates and introducing caching for frequently accessed data like markers and images.

On the backend, I eliminated multiple redundant database writes happening within a single request and ensured updates were handled efficiently.

This reduced unnecessary API calls by ~85%, improved real-time responsiveness and reduced server load significantly, resulting in ~30% cost reduction.

Distributed Data Pipeline (Kafka-based)

The system was originally designed with hardcoded schema-based parsing for incoming NMEA protocol data. Every new data type required manual updates, making the system rigid and difficult to scale.

With thousands of packets arriving per minute, there was also a high risk of data loss and performance bottlenecks.

I took ownership of the problem and redesigned the architecture using Kafka as the core data pipeline.

Instead of hardcoding every data type, I introduced a dynamic parsing mechanism where raw data was stored and processed based on flexible schemas. This removed the need for constant code changes when new data formats were introduced.

This change enabled the system to handle high-frequency data reliably without loss, improved scalability, and simplified future extensibility.

Kafka Node.js MongoDB Event-driven Architecture

Low Latency Video Streaming System

The initial streaming system was unstable and unreliable. Video feeds frequently dropped, reconnection logic was broken, and latency issues made it unusable for real-time monitoring.

Even senior engineers struggled to stabilize the system, and I took ownership of fixing it early in my career.

I worked on stabilizing the entire streaming pipeline by fixing WebRTC signaling, ICE candidate handling, and reconnection flows.

I also integrated AWS Kinesis Video Streams to handle real-time streaming more efficiently and improve reliability.

The final system supported up to 8 concurrent camera feeds with stable performance and significantly improved user experience.

BLE Communication Platform

The application required reliable communication between mobile apps and multiple hardware devices over Bluetooth, which is inherently unstable across platforms.

The initial implementation faced frequent disconnections, inconsistent data transfer and platform-specific issues on Android and iOS.

I redesigned the BLE communication layer by implementing proper connection lifecycle handling, retry mechanisms, and structured message exchange workflows.

I also used native Android and iOS integrations where necessary to ensure consistent behavior across platforms.

This resulted in a stable communication system capable of handling multiple devices with reliable data exchange in real-time.
